# AXIOM v0.9.1 Memory Core Roadmap

## Goal

Define and then implement AXIOM Memory Core after v0.9 final, before Self-Healer work.

## Sequence

1. ADR-004 defines the architecture and laws.
2. Memory Core schema and storage model are added.
3. Memory CRUD and query surfaces are introduced.
4. Memory graph/link semantics are wired into AXIOM core.
5. Export/import is aligned with the `.axiom` exchange format.
6. Optional intelligence layers are added only after the deterministic core is stable.

## In scope for the future implementation PRs

- workspace-scoped memory records
- provenance-aware memory records
- `trust_policy_version` support
- immutable content with supersede links
- tombstone events for deletes
- graph-based memory relationships
- SQL-first persistence
- later Postgres migration path
- CLI, API, MCP, and plugin reuse of the same memory layer

## Out of scope

- separate StackMemory app
- separate StackMemory MCP server
- separate memory database product
- verifier behavior changes
- Trust Kernel bypasses
- release metadata changes
- unrelated runtime refactors

## Suggested implementation order

### PR-M1 - Memory schema and storage

- define memory record shape
- define tombstone and supersede semantics
- add SQL persistence
- keep workspace isolation

### PR-M2 - Memory query surfaces

- `storeMemory`
- `searchMemories`
- `listMemories`
- `queryMemories`
- `timeQuery`

### PR-M3 - Memory graph operations

- `linkMemories`
- `getMemoryGraph`
- graph edges for supersede and provenance relations

### PR-M4 - Batch operations and package exchange

- `batchMemoryOperations`
- `exportMemoryPackage`
- `importMemoryPackage`
- `.axiom` alignment

### PR-M5 - Optional intelligence plugins

- `memorySummary`
- `conceptCluster`
- `embeddingSearch`
- AI digest helpers
- semantic clustering

### PR-M6 - Self-Healer integration

- use memory history for repeat-failure detection
- use reviewed fix facts only
- avoid canonical write shortcuts

## Acceptance criteria

- Memory remains inside AXIOM.
- Memory remains workspace-scoped and provenance-aware.
- Memory does not bypass Trust Kernel validation.
- Deterministic memory features work before any optional intelligence layer is introduced.
